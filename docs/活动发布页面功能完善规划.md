## 二、现状详单：每个页面的功能/字段

### 2.1 活动管理列表 `activity-list.html`

**顶部按钮：** 刷新 / 创建活动（跳 `activity-create.html`）

**筛选：** 活动分类 / 活动状态（已发布/待审核/草稿/已结束/已下线）/ 创建人 / 关键词

**批量操作：** 全选 / 批量提交审核（草稿→待审核）/ 批量发布（待审核→已发布）/ 批量下线（已发布→已下线）/ 批量删除

**表格列：** 复选框 / 活动名称 / 分类 / 状态徽章 / 报名人数（X/Y）/ 活动时间 / 创建人 / 操作

**单行操作（按状态条件渲染）：**
- 查看 → `activity-detail.html?id=`
- 编辑 → `activity-edit.html?id=`
- 草稿 → 提交 / 待审核 → 发布 / 已发布 → 下线 / 任意 → 删除

**分页：** 1-10/共 N 条 + 页码 1、2 + 上一页/下一页

**数据层：** 12 条 Mock 落在 `defaultActivities`；优先读 `sessionStorage.activities_data`，否则首次写入再渲染。

### 2.2 创建活动 `activity-create.html`

**表单 Tab：**
- 基础信息（默认）
- 活动详情（textarea + 假工具栏）
- 资源引用（图片/文件/视频 三子 Tab）

**基础信息字段：**
- 活动名称* / 活动分类* / 标签（逗号分隔）/ 活动封面（资源中心选 + 本地双入口）/ 活动简介 / 活动地点 / 主办单位 / 最大报名人数
- 活动时间（开始-结束 datetime-local）/ 报名时间（开始-结束 datetime-local）
- 发布形式：存为草稿 / 提交审核 / 直接发布

**活动详情 Tab：** 富文本 textarea + 工具栏（加粗/斜体/下划线/无序列表/有序列表/链接/插入图片/插入资源）—— 工具栏均为按钮样式，无实际富文本逻辑。

**资源引用 Tab：** 3 个子 Tab，搜索+点击切换"已选"，底部"已选资源"展示区，可移除。

**底部按钮：** 取消 / 暂存（draft）/ 提交（pending）；保存 toast 后 setTimeout 跳回列表。

**校验：** 仅校验活动名称、分类必填；其他字段无校验；时间区间无校验。

### 2.3 编辑活动 `activity-edit.html`

与创建页几乎完全相同，差异：
- URL 必须带 `?id=`，否则报错跳列表
- 加载时从 sessionStorage 取数据回填
- `publishType` 单选框回填：published→published、pending→pending、其他→draft
- 缺少封面回填逻辑（list Mock 无 `coverUrl`，edit 页能存但 list 不写）

### 2.4 活动详情 `activity-detail.html`

**顶部：** 封面图（h-64）+ 状态徽章浮层 + 标题 + 元信息行 + 标签 + 返回/编辑按钮

**Tab：** 基础信息 / 活动详情 / 资源引用

**基础信息 Tab：** 活动名称、分类、地点、主办、最大报名人数、已报名人数、活动时间区间、报名时间区间、标签、简介、创建人、发布时间

**活动详情 Tab：** 纯文本展示（whitespace-pre-wrap）

**资源引用 Tab：** 网格 5 列展示（图片/文件/视频占位）

### 2.5 活动分类 `activity-category.html`

表格列：序号 / 分类名称 / 分类编码 / 排序 / 状态徽章 / 描述 / 操作（编辑/启用切换/删除）

新增/编辑弹窗：分类名称*、分类编码*（创建必填，编辑只读）、排序、描述

校验：分类编码唯一性

数据：7 条默认 Mock（tech/culture/sports/education/charity/summit/team）

### 2.6 活动审核 `activity-review.html`

## 三、缺失功能详单

### 3.1 核心业务闭环缺失

| # | 缺失项 | 影响 |
|---|---|---|
| C1 | **活动发布独立页** | 菜单注册 `activityPublish → content-publish.html`，实际是内容发布复用页，活动无专属发布流程 |
| C2 | **审核结果不回写 list** | `activity-review.html` 用独立 Mock，通过/驳回不写回 `activities_data`，list 看不到审核结果 |
| C3 | **list 发布绕过审核** | list 可对 pending 直接 publish，越权操作 |
| C4 | **报名审批不更新 participants** | `activity-registration.html` 通过后 activity.participants 不变 |
| C5 | **签到与报名不通** | `activity-signin.html` 的 activityData 与 `activity-registration.html` 完全隔离 |
| C6 | **无活动自动结束机制** | endTime 到后 status 不会自动转 ended |
| C7 | **已驳回活动无回退入口** | list 中 rejected 状态无"重新提交审核"按钮 |
| C8 | **已下线活动无恢复入口** | list 中 offline 状态无"重新上线"按钮 |
| C9 | **无活动归档** | 无 archived 状态，无归档页 |
| C10 | **用户端数据完全独立** | `portal/index.html` 用自己 mockData，不读后台任何数据；list/create/edit 写入的 key 与 portal 读取的 key 不一致 |

## 四、逻辑闭环分析

### 4.1 理想闭环链路

```
创建（create） → 提交（pending） → 审核（review） → 通过/驳回（approved/rejected）
   → 发布（published）→ 用户端展示 → 用户报名 → 报名审核 → 签到
   → 活动结束（auto ended）→ 数据统计 → 归档（archived）
```

### 4.2 实际闭环链路

```
创建 → [提交] → 待审核（写入 sessionStorage.activities_data 的 status=pending）
        ↓
审核（独立 Mock）→ [通过/驳回]（仅更新 review 内部数组，不写回 activities_data）
        ↓
list → [发布]（手动从 pending→published，绕过 review 页）
        ↓
发布后 → 用户端 activity-zone（完全独立的 mockData，不读后台）
        ↓
报名管理 → 仅查看/审批（不写回活动 participants）
        ↓
签到（独立数据，与报名库不通）
        ↓
统计（完全硬编码）
        ↓
归档（无）
```

### 4.3 闭环断裂点（8 处）

| # | 断裂点 | 现状 | 影响 |
|---|---|---|---|
| B1 | **review → list** | 审核结果不回写 | list 看不到审核结果，运营需手动发布 |
| B2 | **list 发布绕过审核** | list 可对 pending 直接 publish | 流程不规范 |
| B3 | **publish → 用户端** | portal/activity-zone 完全用自己 mockData | 后台发布后用户看不到 |
| B4 | **报名审批 → 活动 participants** | 审批通过不更新活动报名数 | 报名数永远不准 |
| B5 | **报名 → 签到候选** | signin 的 activityData 与 registration 隔离 | 签到名单是凭空生成的 |
| B6 | **签到 → 活动状态** | 签到不影响活动 status | 活动结束后还能签到 |
| B7 | **stats → list** | stats 完全静态 | 报表数据无法反映真实业务 |
| B8 | **归档** | 无任何归档入口 | 历史活动堆积，难以管理 |

### 4.4 端到端数据流缺失图

```
后台创建活动 (activity-create)
    │
    ↓ 写入 sessionStorage.activities_data
    │
[后台审核/发布/下线/删除]  ←── ✅ 内部闭环
    │
    ↓ (断裂点 B3) 不传向用户端
    │
portal/index.html & activity-zone/index.html
    │
    ↓ 使用完全独立的 mockData
    │
[用户端报名/签到]
    │
    ↓ (断裂点 B4/B5) 不传回后台
    │
后台报名管理/签到管理 ←── 独立 Mock 数据
    │
    ↓ (断裂点 B7) 不联动统计
    │
活动统计 ←── 完全静态硬编码
```

## 八、技术要点与风险

### 8.1 数据迁移策略

**问题：** 当前 4 套数据源（sessionStorage / localStorage / 硬编码 / portal mockData）需要合并。

**策略：**
1. 第一次加载时按优先级合并：localStorage（新） > sessionStorage（旧） > 默认 Mock
2. 迁移脚本在 `components/data-store.js` 的 `init()` 中执行
3. 数据版本号 `__version__` 字段控制 schema 演进

### 8.2 性能与状态机

**问题：** 8 个页面共享同一份数据，每次操作都要重渲染。

**策略：**
1. 引入极简发布订阅：`dataStore.subscribe(key, callback)`
2. 每次写操作触发订阅，回调中按需重渲染
3. 避免全表重绘：用 `data-id` 选择器精确更新

### 8.3 用户端与后台数据同步

**问题：** portal/index.html 是 SPA，但与后台 list 不在同一域名（实际同域，但是不同页面）。

**策略：**
1. localStorage 天然支持跨标签页同步
2. 监听 `storage` 事件：后台修改后用户端自动刷新
3. 或者在 portal 切回时强制 reload

### 8.4 富文本编辑器选型

**问题：** textarea 无法满足真实富文本需求。

**策略：**
1. 阶段一保持 textarea（仅贯通闭环）
2. 阶段三再升级：候选 wangEditor 5（~150KB）/ TinyMCE 6（~400KB）/ Quill 2（~200KB）

## 十、风险与约束

| 风险 | 影响 | 缓解 |
|---|---|---|
| localStorage 容量限制 | 5-10MB，活动数量大时可能不够 | 长期方案：分页 + 归档分离 |
| 状态机重构导致旧数据失效 | 用户现有 sessionStorage 数据丢失 | 数据迁移脚本兼容旧 key |
| 富文本升级引入复杂度 | 增加维护成本 | 阶段三再升级，先保持 textarea |
| H5 签到依赖 HTTPS | 原型阶段无法真实扫码 | 阶段四增加手动输入兜底 |
| portal 跨标签页同步 | 后台修改用户端可能不刷新 | storage 事件 + 切回强制 reload |
